## 决策 001：机票预订的“四步向导”与库存锁定模型

机票订购流程尽可能模仿携程的4步向导式订购流程，每一个页面代表机票的不同订购步骤。所有的数据变动关键是订单状态转换。

### 1. 上下文 (Context)

为实现机票预订功能，我们面临两个关键挑战：
1.  **流程设计：** 如何设计一个既能模仿携程多步骤体验，又符合Next.js App Router特性的预订流程。
2.  **库存管理：** 如何在多用户并发场景下，正确处理稀缺的座位库存，即如何解决“竞态条件 (Race Condition)”。

### 2. 决策 (Decision)

我们决定采用一个**基于URL的、四步向导式**的预订流程，并配合**“悲观锁”**模型在**第一步结束时**创建订单并锁定库存。

### 竞态条件

主体采用悲观锁。我们平台和携程平台的最大区别在于携程是OTA平台，并不负责提供机票数据，而是通过外部数据源，也就是携程只负责提供乘机人等等信息以及转发功能，然后根据外部数据员的反馈相应地更新自己的数据库或订单等等。但是我们的 Nomad 项目的特殊之处在于我们既提供 OTA 功能，也提供机票数据（尽管是 Mock的）。

虽然这个是课程项目，但是我认为完全不考虑 race condition是不可以接受的。所谓 race condition， 可以考虑如下场景：某一个航班仓库中仅剩一个余票，两个用户的订票请求**同时**到达数据库，如果不妥善处理，可能出现数据库先处理A的请求，发现余票为1，因此允许给A卖票，此时B的查询也到达，因为A的买票还没有完全处理完成，因此B也发现1张票，也被允许购票，此时**“超售”**。

当然也不是锁定的越早越好，这容易导致大量座位被空锁定。并且从实现的角度这也是不可能的，携程平台在机票搜索的时候不支持人数选择（尽管移动端上似乎支持这个功能，但是 Nomad 不会实现这个功能），因此我们无法在用户选择乘机人（可能选择1个或多个）之前知道需要售卖几张机票，因此也就无法锁定库存。

我们进一步探索携程，在第一步结束，进入第二步之后，我们就可以在“我的订单”中查看创建的带支付的订单，同时能看到支付倒计时。因此我们选择在第一步成功后创建订单锁定库存。

当然需要注意这个相应的代价就是可能存在用户A先点击了订票按钮，但是一直没有完成乘机人的填写，而此时用户B购买了所有剩余的票，等到A填写完成并提交的时候，会检查库存发现不足并拒绝A的订单创建。我们认为这个UX上的损失是可以接受的。

从实现的角度就是需要使用数据库的事务。这里不具体设计实现方式，但是一定要确保检查库存，和扣减库存创建订单是在一个事务内部的。同时事务不能过分庞大复杂，将不需要逻辑放到事务外边以防长时间占用数据库。

### 清理过时订单

必须暴露一个API负责让外部cron job调用，以清理过时订单。

可选的方案有vercel cron job，和cron-job.com等等。

尽管我们的demo网站就部署在 vercel 平台上，但是vercel的free tier

同样的 github action + schedule 有类似的问题。尽管 public repository可以近似不限量地运行workflow，但是官方文档明确提到不能保证准时运行，尤其是在整点等等时候可能因为服务器压力过大而排队运行，这对于我们比较短的支付窗口是不能接受的。

目前暂时决定采用 cron-job。待定。似乎是免费的。
