## 文档说明

本文档记录 OTA 平台开发中常见的业务安全漏洞，适用于：

- **自查本组代码**：确保核心业务逻辑安全
- **攻防环节测试其他组**：快速定位常见漏洞

漏洞按风险等级分为 P0（必考虑）、P1（应考虑）、P2（超纲）三级。

---

## P0 级漏洞

> 基础安全问题，任何 OTA 平台都应覆盖

### 1. 价格篡改漏洞

**业务场景：** 用户提交订单、支付流程  
**漏洞描述：** 前端传递价格参数到后端，后端未校验直接使用

**测试方法：**

1. 浏览器开发者工具拦截支付请求
2. 修改 `totalPrice` 字段为 `0.01`
3. 提交请求，观察是否能以极低价格完成支付

**正确实现：**

- ✅ 后端根据航班ID、舱位、乘客数量**重新计算价格**
- ✅ 忽略前端传来的价格参数，仅用于前端显示对比验证
- ❌ 直接使用 `request.body.price` 写入订单

**测试数据：**

```json
// 恶意请求示例
POST /api/orders
{
  "flightId": "CA1234",
  "passengers": ["passenger1", "passenger2"],
  "totalPrice": 1  // 篡改为 1 元
}
```

### 2. 横向越权漏洞 (IDOR)

**业务场景：** 常旅客管理、订单查看/修改/删除  
**漏洞描述：** 用户A通过修改请求参数访问/修改用户B的数据

**测试方法：**

1. 用户A登录，查看自己的常旅客列表，记录某个旅客ID（如 `passenger_123`）
2. 用户B登录，创建自己的常旅客，记录ID（如 `passenger_456`）
3. 用户A修改请求，将 `passenger_123` 改为 `passenger_456`，尝试删除
4. 检查是否能成功删除用户B的旅客

**正确实现：**

- ✅ 后端在所有资源操作前验证 `资源.userId === 当前登录用户ID`
- ✅ 查询数据库时同时过滤 `WHERE id = ? AND user_id = ?`
- ❌ 仅验证登录态，不验证资源所属关系

**测试场景：**

- [ ] 删除他人的常旅客信息
- [ ] 查看他人的订单详情
- [ ] 取消他人的订单
- [ ] 修改他人的个人信息

### 3. 后端输入验证缺失

**业务场景：** 所有接受用户输入的接口  
**漏洞描述：** 后端未校验输入合法性，可能导致数据污染或注入攻击

**测试方法：**

1. 在姓名字段输入超长字符串（如 10000 个 'A'）
2. 在数字字段输入负数、浮点数、极大值
3. 在必填字段传 `null` 或空字符串
4. 在枚举字段传非法值（如性别传 `attack`）

**测试数据：**

```json
// 异常输入测试
{
  "chineseName": "A".repeat(10000),  // 超长字符串
  "age": -999,                        // 负数
  "gender": "'; DROP TABLE users; --", // SQL 注入尝试
  "documentNumber": "<script>alert('xss')</script>" // XSS 尝试
}
```

**正确实现：**

- ✅ 后端使用验证库（如 Zod、Joi、class-validator）严格校验
- ✅ 定义字段长度、格式、范围限制
- ❌ 信任前端验证，后端直接入库

### 4. SQL 注入漏洞

**业务场景：** 搜索、筛选功能  
**漏洞描述：** 用户输入直接拼接到 SQL 语句中

**测试方法：**

1. 在航班搜索框输入：`上海' OR '1'='1`
2. 在订单号搜索输入：`123'; DROP TABLE orders; --`
3. 观察是否返回异常数据或触发错误

**正确实现：**

- ✅ 使用 ORM 框架或参数化查询
- ✅ 示例（参数化）：`SELECT * FROM flights WHERE departure = ?`
- ❌ 字符串拼接：`SELECT * FROM flights WHERE departure = '${userInput}'`

### 5. 认证与会话安全

**业务场景：** 登录、注册、会话管理  
**漏洞描述：** 未登录用户可访问受保护资源，或会话可被劫持

**测试方法：**

1. 未登录状态下直接访问 `/api/orders/create`
2. 复制已登录用户的 Cookie 到其他浏览器，观察是否能直接使用
3. 修改 Cookie 中的 `userId` 字段，尝试伪造身份
4. 登出后使用旧 Token，观察是否仍有效

**测试场景：**

- [ ] 未登录访问订单列表
- [ ] 未登录删除常旅客
- [ ] 伪造 userId 访问他人数据
- [ ] 登出后 Token 未失效

**正确实现：**

- ✅ 所有写操作接口必须验证登录态
- ✅ 会话 Token 加密存储，不可轻易伪造
- ✅ 登出后使 Token 立即失效

### 6. 敏感信息泄露

**业务场景：** API 响应、日志记录、前端页面  
**漏洞描述：** 返回不必要的敏感字段，或日志记录明文密码

**测试方法：**

1. 查看登录接口返回值，检查是否包含 `password` 字段
2. 查看用户列表接口，检查是否返回完整身份证号、手机号
3. 故意触发错误，检查错误信息是否暴露数据库结构

**测试场景：**

- [ ] 登录接口返回密码哈希
- [ ] 用户信息接口返回完整身份证号（应脱敏为 `320***********1234`）
- [ ] 错误信息暴露表名、字段名
- [ ] 前端代码包含 API 密钥或数据库密码

**正确实现：**

- ✅ API 响应仅返回必要字段
- ✅ 敏感字段脱敏：`***`、`320***1234`
- ✅ 生产环境关闭详细错误堆栈

### 货币/价格计算

**业务场景：** 任何涉及货币计算的地方。
* **后端：** API 响应、订单总价计算、税费/折扣计算、数据库存储。
* **前端：** 购物车总价计算、分期付款显示、价格相加。
* **日志：** 记录的交易金额。

**漏洞描述：**
在计算机中，标准的 `float` 或 `double` 类型（在 JavaScript 中就是 `number` 类型）是**二进制浮点数**，它们无法精确表示 `0.1` 这样的十进制小数。

在任何涉及货币的地方（尤其是加法、乘法、除法）使用这些不精确的类型，会**必然**导致微小的舍入误差（例如 `0.1 + 0.2` 变成 `0.30000000000000004`）。这些微小的误差在多次计算后会累积，导致订单总价错误、支付金额与账单不符、或退款金额计算错误，是**灾难性**的财务 Bug。

**测试方法：**

1.  **（黑盒）输入特定值：** 尝试输入或组合会暴露浮点数问题的特定价格（见下方“测试场景”），尤其是包含 `.1`, `.2`, `.3`, `.6`, `.7`, `.9` 等小数的。
2.  **（白盒）检查数据库 Schema：** 查看项目的数据库设计文档或 `schema` 文件。如果任何存储货币的字段使用了 `float`, `double precision`, 或 `real` 类型，这**几乎可以 100% 确定**存在此漏洞。
3.  **（白盒）审查代码：**
    * **检查 `parseFloat` / `Number()`：** 检查后端代码是否在计算时使用了 `parseFloat()` 或 `Number()` 将价格转为 JavaScript `number` 类型。
    * **检查整数百分比计算：** 如果对方使用了“分单位”存储法，检查他们在计算百分比（如7%的税）时，是否正确处理了中间除法（`priceInCents * 7 / 100`）导致的浮点数和舍入问题。

**测试场景：**

- [ ] **加法测试：** 在购物车或订单中，添加两个价格难以精确表示的商品（例如 `¥10.10` 和 `¥20.20`），检查总价是否为精确的 `¥30.30`（而不是 `30.3000...04`）。
- [ ] **乘法测试：** 购买 **3** 件单价为 `¥9.99` 的商品。检查总价是否为精确的 `¥29.97`。
- [ ] **除法/百分比测试：** 对一个 `¥123.45` 的订单，应用一个 10% 的折扣。检查折扣金额是否被正确地舍入到“分”单位（例如，是 `¥12.35` 还是 `¥12.34`？），并验证最终总价是否正确。
- [ ] **累积误差测试：** 连续添加 10 件 `¥0.1` 的商品，检查总价是否为 `¥1.00`。

**正确实现：**

- **方案 A：使用定点十进制（Decimal / Numeric）类型**
    * **数据库：** 在 Schema 中使用了 `DECIMAL(P, S)` 或 `NUMERIC(P, S)` 类型（例如 `NUMERIC(10, 2)`）。
    * **后端代码：** 在应用层**从不**将此类型读取为标准浮点数。而是将其作为 **`string`** 或 **`BigDecimal` / `Decimal.js`** 等高精度库对象来处理所有数学运算。

- **方案 B：使用整数（分单位）存储**
    * **数据库：** 在 Schema 中使用了 `INTEGER` 或 `BIGINT` 类型来存储所有价格（例如，`¥123.45` 存储为整数 `12345`）。
    * **后端代码：** 所有的数学运算**全部使用整数算术**。只有在**最后一步**（即返回给前端展示或API响应时）才执行 `/ 100` 操作将其转换为元单位。

## P1 级漏洞

> 进阶问题，优秀作业应覆盖

### 7. 订单并发与库存竞争

**业务场景：** 仅剩1张票，两个用户同时预订  
**漏洞描述：** 未使用事务或锁机制，导致超卖

**测试方法：**

1. 修改数据库，将某航班剩余座位数改为 1
2. 开启两个浏览器标签，同时点击"立即预订"
3. 检查是否生成了 2 个订单，剩余座位变为 -1

**压力测试脚本（伪代码）：**

```bash
# 使用 Apache Bench 或类似工具
ab -n 10 -c 10 http://localhost:3000/api/book-flight?id=CA1234
# 10 个并发请求同时预订同一航班
```

**正确实现：**

- ✅ 使用数据库事务 + 悲观锁（`SELECT ... FOR UPDATE`）
- ✅ 扣减库存时检查：`UPDATE seats SET remaining = remaining - 1 WHERE id = ? AND remaining >= 1`
- ❌ 先查询库存，再单独执行扣减（非原子操作）

### 8. 订单状态机漏洞

**业务场景：** 订单状态流转（待支付 → 待出行 → 已完成）  
**漏洞描述：** 未校验状态转换合法性，可直接跳过支付

**测试方法：**

1. 创建订单，状态为"待支付"
2. 拦截"取消订单"请求，修改为"标记为已完成"
3. 观察是否能直接将待支付订单改为已完成

**合法状态转换表：**

```
待支付 → 已取消 ✅
待支付 → 待出行 ✅（支付后）
待支付 → 已完成 ❌（非法）
待出行 → 已完成 ✅
已取消 → 待出行 ❌（非法）
```

**测试数据：**

```json
// 恶意请求
PATCH /api/orders/123
{
  "status": "COMPLETED"  // 尝试跳过支付
}
```

### 9. 支付倒计时校验缺失

**业务场景：** 订单支付超时自动取消  
**漏洞描述：** 仅前端显示倒计时，后端未校验，用户可无限期支付

**测试方法：**

1. 创建订单，等待超过 30 分钟
2. 刷新页面前端倒计时已归零，但仍尝试支付
3. 观察是否能成功支付过期订单

**正确实现：**

- ✅ 后端定时任务每分钟扫描超时订单，自动取消
- ✅ 支付接口校验：`IF order.createdAt + 30min < NOW() THEN REJECT`
- ❌ 仅前端禁用支付按钮

### 10. 频率限制缺失

**业务场景：** 短信验证码、登录、搜索  
**漏洞描述：** 未限制请求频率，可被暴力破解或 DDoS

**测试方法：**

1. 编写脚本循环发送验证码请求
2. 观察是否能在短时间内发送 100+ 条验证码
3. 尝试暴力破解 OTP：循环尝试 `000000` ~ `999999`

**测试脚本：**

```python
import requests
for i in range(1000):
    requests.post("http://target.com/api/send-otp", json={"phone": "13800138000"})
```

**正确实现：**

- ✅ 同一 IP/用户 1 分钟内最多 5 次请求
- ✅ 同一手机号 1 小时内最多 3 次验证码
- ✅ OTP 输错 3 次后锁定 10 分钟

### 11. OTP 验证码安全

**业务场景：** 手机/邮箱验证码登录  
**漏洞描述：** 验证码可重复使用、无过期时间、易被暴力破解

**测试方法：**

1. 获取验证码 `123456`，使用后再次提交相同验证码
2. 等待 1 小时后使用旧验证码
3. 尝试 6 位数字暴力破解（`000000` ~ `999999`）

**正确实现：**

- ✅ 验证码 5 分钟内有效
- ✅ 使用后立即失效（标记为已使用）
- ✅ 错误尝试 3 次后锁定
- ✅ 使用 6 位数字 + 大小写字母（增加复杂度）

### 12. 退款逻辑漏洞

**业务场景：** 订单退款  
**漏洞描述：** 未校验订单状态，可重复退款或退款未支付订单

**测试方法：**

1. 对"待支付"订单发起退款请求
2. 对同一"已支付"订单多次点击退款
3. 观察账户余额是否被多次退款

**正确实现：**

- ✅ 仅"待出行"状态的订单可退款
- ✅ 退款后订单状态改为"已退款"（终态）
- ✅ 使用事务确保退款操作原子性

## P2 级漏洞

> 超纲内容，作为加分项

### 13. CSRF 攻击

**漏洞描述：** 用户在已登录状态下访问恶意网站，恶意网站伪造请求  
**测试方法：** 构造恶意 HTML 页面，自动提交表单到目标网站

**防护措施：**

- ✅ 使用 CSRF Token
- ✅ 验证 `Referer` 或 `Origin` 头
- ✅ 关键操作使用 POST 而非 GET

---

### 14. XSS 攻击

**漏洞描述：** 用户输入的恶意脚本被渲染到页面  
**测试数据：**

```html
<script>
  alert(document.cookie);
</script>
<img src="x" onerror="alert('XSS')" />
```

**防护措施：**

- ✅ 前端框架（React/Vue）默认转义
- ✅ 后端存储前过滤危险标签
- ✅ 设置 `Content-Security-Policy` 头

---

### 15. 敏感数据加密存储

**漏洞描述：** 身份证号、护照号明文存储，数据库泄露后直接暴露

**正确实现：**

- ✅ 使用 AES-256 加密存储
- ✅ 密钥存储在环境变量，不提交到 Git
- ❌ 明文存储或使用 MD5（不可逆）
