## 分支策略

我们采用基于 **Git Flow** 简化版的分支管理策略，主要包含以下分支类型：

### 主要分支

#### `main` 分支

- **用途**：生产环境代码，始终保持稳定可发布状态
- **保护规则**：
  - 禁止直接推送
  - 必须通过 Pull Request 合并
  - 需要至少 1 人审查批准
  - 必须通过所有 CI 检查
- **合并来源**：仅接受来自 `develop` 分支的合并

#### `develop` 分支

- **用途**：开发主分支，包含最新的开发进度
- **保护规则**：
  - 禁止直接推送
  - 必须通过 Pull Request 合并
  - 需要通过 CI 检查
- **合并来源**：接受来自功能分支、修复分支的合并

### 临时分支

<Tabs items={['功能分支', '修复分支', '热修复分支']}>
  <Tab value="功能分支">
    **命名规范**：`feat/功能描述`

    **示例**：
    - `feat/user-authentication`
    - `feat/flight-search`
    - `feat/payment-integration`

    **生命周期**：
    1. 从 `develop` 分支创建
    2. 开发完成后合并回 `develop`
    3. 合并后删除

    **适用场景**：开发新功能

  </Tab>

  <Tab value="修复分支">
    **命名规范**：`fix/问题描述`

    **示例**：
    - `fix/login-button-styling`
    - `fix/search-validation`
    - `fix/payment-timeout`

    **生命周期**：
    1. 从 `develop` 分支创建
    2. 修复完成后合并回 `develop`
    3. 合并后删除

    **适用场景**：修复开发环境中的 bug

  </Tab>

  <Tab value="热修复分支">
    **命名规范**：`hotfix/紧急问题描述`

    **示例**：
    - `hotfix/critical-security-patch`
    - `hotfix/payment-failure`

    **生命周期**：
    1. 从 `main` 分支创建
    2. 修复完成后同时合并回 `main` 和 `develop`
    3. 合并后删除

    **适用场景**：修复生产环境的紧急问题

  </Tab>
</Tabs>

## Commit 提交规范

我们严格遵循 [Conventional Commits](https://www.conventionalcommits.org/) 规范，并通过 `commitlint` 自动检查。

### 提交信息格式

```
<type>(<scope>): <subject>

<body>

<footer>
```

### Type 类型说明

<Tabs items={['必须了解', '常用类型', '其他类型']}>
  <Tab value="必须了解">
    | Type | 说明 | 示例 |
    |------|------|------|
    | `feat` | 新功能 | `feat(auth): add GitHub OAuth login` |
    | `fix` | Bug 修复 | `fix(search): resolve date validation issue` |
    | `docs` | 文档变更 | `docs: update git workflow guide` |
  </Tab>

<Tab value="常用类型">
  | Type | 说明 | 示例 | |------|------|------| | `style` |
  代码格式（不影响功能） | `style: format code with prettier` | | `refactor` |
  重构 | `refactor(api): simplify error handling` | | `test` | 测试相关 | `test:
  add unit tests for utils` | | `chore` | 其他（构建、工具等） | `chore: update
  dependencies` |
</Tab>

  <Tab value="其他类型">
    | Type | 说明 | 示例 |
    |------|------|------|
    | `perf` | 性能优化 | `perf: optimize database queries` |
    | `ci` | CI/CD 配置 | `ci: add build workflow` |
    | `build` | 构建系统 | `build: update webpack config` |
    | `revert` | 回滚 | `revert: revert commit abc123` |
    | `wip` | 开发中（不推荐提交） | `wip: working on feature` |
  </Tab>
</Tabs>

### Scope 范围（可选）

Scope 用于指定提交影响的范围，常用的 scope 包括：

- `auth` - 认证相关
- `ui` - 用户界面
- `api` - API 接口
- `db` - 数据库
- `utils` - 工具函数
- `config` - 配置文件
- `deps` - 依赖管理

### Subject 主题规则

<Callout title="主题编写规范" type="warn">
  - ✅ 使用祈使句，首字母小写 - ✅ 不超过 100 字符 - ✅ 末尾不加句号 - ❌
  避免使用 "added"、"updated" 等过去式 - ❌ 不要使用
  sentence-case、start-case、pascal-case 或 upper-case
</Callout>

## 开发工作流程

### 完整开发流程

<Steps>
<Step>
### 创建功能分支

从最新的 `develop` 分支创建新分支：

```bash
# 确保 develop 分支是最新的
git checkout develop
git pull origin develop

# 创建并切换到新分支
git checkout -b feat/your-feature-name
````

</Step>

<Step>
### 开发和提交

在功能分支上进行开发，并遵循提交规范：

```bash
# 查看修改
git status

# 添加文件到暂存区
git add .

# 提交（会自动触发 lint-staged 和 commitlint）
git commit -m "feat(module): add new feature"
```

<Callout title="自动化检查" type="info">
提交时会自动运行：
- **lint-staged**：对暂存文件运行 ESLint 和 Prettier
- **commitlint**：检查提交信息是否符合规范

如果检查失败，提交会被阻止，需要修复问题后重新提交。

</Callout>
</Step>

<Step>
### 推送到远程

```bash
# 首次推送需要设置上游分支
git push --set-upstream origin feat/your-feature-name

# 后续推送
git push
```

</Step>

<Step>
### 创建 Pull Request

1. 在 GitHub 上创建 Pull Request
2. 填写 PR 模板中的必要信息
3. 关联相关的 Issue（如果有）
4. 等待 CI 检查通过
5. 请求团队成员审查

<Callout title="PR 模板" type="info">
项目提供了 PR 模板（`.github/pull_request_template.md`），包含：
- 变更描述
- 相关 Issue
- 变更类型
- 测试清单
- 代码审查检查清单
</Callout>
</Step>

<Step>
### 代码审查

审查者需要检查：

- 代码质量和可读性
- 是否有适当的测试
- 是否有性能问题
- 是否有安全隐患
- 破坏性变更是否有文档说明

</Step>

<Step>
### 合并和清理

PR 被批准并通过所有检查后：

```bash
# 在 GitHub 上合并 PR（推荐使用 Squash and merge）

# 切换回 develop 分支
git checkout develop

# 拉取最新代码
git pull origin develop

# 删除本地功能分支
git branch -d feat/your-feature-name

# 删除远程功能分支（如果 GitHub 没有自动删除）
git push origin --delete feat/your-feature-name
```

</Step>
</Steps>

## Git Hooks 自动化

项目使用 **Husky** 管理 Git Hooks，在特定的 Git 操作时自动执行检查。

### Pre-commit Hook

在每次提交前自动运行 `lint-staged`：

```bash title=".husky/pre-commit"
pnpm lint-staged
```

**lint-staged 配置**（`package.json`）：

```json
{
  "lint-staged": {
    "*.{js,jsx,ts,tsx}": ["eslint --fix", "prettier --write"],
    "*.{json,md,mdx,css,html,yml,yaml}": ["prettier --write"]
  }
}
```

<Callout title="自动修复" type="info">
  lint-staged
  会自动修复可以修复的问题（如代码格式），并将修复后的文件添加到暂存区。如果有无法自动修复的问题，提交会被阻止。
</Callout>

### Commit-msg Hook

在提交信息创建后验证其格式：

```bash title=".husky/commit-msg"
pnpm commitlint --edit $1
```

**commitlint 配置**（`commitlint.config.mjs`）：

```javascript
export default {
  extends: ["@commitlint/config-conventional"],
};
```

### 提交信息模板

项目提供了提交信息模板（`.gitmessage`），可以通过以下命令配置：

```bash
git config commit.template .gitmessage
```

配置后，每次运行 `git commit` 时会自动加载模板，提供提交规范的提示。

## CI/CD 自动化

项目使用 **GitHub Actions** 实现持续集成和持续部署。

### 工作流概览

| 工作流               | 触发条件                    | 主要任务          |
| -------------------- | --------------------------- | ----------------- |
| **Build Check**      | PR 到 `main`/`develop`      | 构建检查          |
| **Unit Tests**       | Push/PR 到 `main`/`develop` | 单元测试 + 覆盖率 |
| **Playwright Tests** | Push/PR 到 `main`/`develop` | E2E 测试          |
| **Deploy**           | Push 到 `main`              | 部署到生产环境    |

### Build Check 工作流

<Tabs items={['配置文件', '执行步骤', '失败处理']}>
  <Tab value="配置文件">
```yaml title=".github/workflows/build.yml"
name: PR Build Check

on:
pull_request:
branches: [main, develop]
types: [opened, synchronize, reopened]

jobs:
build:
runs-on: ubuntu-latest
steps: - name: Checkout code
uses: actions/checkout@v4

      - name: Install pnpm
        uses: pnpm/action-setup@v2
        with:
          version: 10

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: "pnpm"

      - name: Install dependencies
        run: pnpm install --frozen-lockfile

      - name: Run build
        run: pnpm build

````
  </Tab>

  <Tab value="执行步骤">
**执行步骤**：

1. **检出代码**：获取 PR 的最新代码
2. **安装 pnpm**：使用 pnpm 10.x 版本
3. **设置 Node.js**：使用 Node.js 20.x 版本
4. **安装依赖**：使用 `--frozen-lockfile` 确保依赖版本一致
5. **运行构建**：执行 `pnpm build` 检查是否能成功构建

<Callout title="构建环境变量" type="warn">
构建过程需要一些环境变量（如 `DATABASE_URL`），CI 中使用 dummy 值仅用于构建检查，不会连接真实数据库。
</Callout>
  </Tab>

  <Tab value="失败处理">
如果构建失败，按以下步骤处理：

**排查步骤**：
1. 查看日志 - 点击 GitHub Actions 中的失败任务查看详细日志
2. 本地复现 - 在本地运行 `pnpm build` 复现问题
3. 修复问题 - 根据错误信息修复代码
4. 重新提交 - 提交修复后会自动重新运行 CI

**常见构建失败原因**：TypeScript 类型错误、ESLint 错误、导入路径错误、缺少必要的环境变量
  </Tab>
</Tabs>

### Unit Tests 工作流

```yaml title=".github/workflows/test.yml"
name: Unit Tests

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main, develop]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Install dependencies
        run: pnpm install --frozen-lockfile

      - name: Run unit tests
        run: pnpm test:coverage

      - name: Upload coverage to Codecov
        uses: codecov/codecov-action@v4
        with:
          file: ./coverage/lcov.info
````

**特性**：

- 自动运行所有单元测试
- 生成测试覆盖率报告
- 上传覆盖率到 Codecov
- 在 PR 中自动评论测试结果

### Playwright E2E Tests 工作流

```yaml title=".github/workflows/playwright.yml"
name: Playwright Tests

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main, develop]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Install dependencies
        run: pnpm install --frozen-lockfile

      - name: Install Playwright browsers
        run: pnpm exec playwright install --with-deps

      - name: Run Playwright tests
        run: pnpm e2e

      - name: Upload test results
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: playwright-report
          path: playwright-report/
```

**特性**：

- 自动安装浏览器
- 运行所有 E2E 测试
- 保存测试报告和视频
- 失败时保留 artifacts 供调试

### 解决冲突

```bash
# 1. 拉取最新代码时发生冲突
git pull origin develop
# 手动解决冲突文件中的标记
# <<<<<<< HEAD
# 你的修改
# =======
# 远程的修改
# >>>>>>> commit-hash

# 2. 标记冲突已解决

# 3. 完成合并
git commit

# 4. 如果是 rebase 过程中的冲突
git add <resolved-file>
git rebase --continue

# 5. 放弃解决冲突
git merge --abort  # 或
git rebase --abort
```

## 最佳实践

### 提交粒度

<Callout title="提交原则" type="info">
  - ✅ **小而频繁**：每个提交应该是一个逻辑上完整的改动 - ✅
  **单一职责**：一个提交只做一件事 - ✅
  **可回滚**：每个提交都应该是可以安全回滚的 - ❌
  **避免大提交**：不要在一个提交中包含多个不相关的改动 - ❌
  **避免半成品**：不要提交未完成或会破坏构建的代码
</Callout>

**好的提交示例**：

```bash
# 每个提交都是独立的功能点
git commit -m "feat(auth): add login form UI"
git commit -m "feat(auth): implement login API integration"
git commit -m "feat(auth): add form validation"
git commit -m "test(auth): add login form tests"
```

**不好的提交示例**：

```bash
# 一个提交包含太多内容
git commit -m "feat: add login, fix bugs, update docs"
```

### 分支管理

1. **及时同步**：定期从 `develop` 同步最新代码到功能分支

   ```bash
   git fetch origin
   git rebase origin/develop
   ```

2. **保持分支整洁**：合并后及时删除已完成的分支

   ```bash
   git branch -d feat/completed-feature
   ```

3. **避免长期分支**：功能分支不应该存在超过 1-2 周，应该拆分成更小的功能

4. **使用描述性名称**：分支名应该清楚地表达其目的
   - ✅ `feat/user-profile-page`
   - ❌ `my-branch` 或 `test`

### Pull Request 最佳实践

<Steps>
<Step>
### 创建前的准备

- 确保代码已经过本地测试
- 运行 `pnpm lint` 和 `pnpm test` 确保没有错误
- 从 `develop` 同步最新代码并解决冲突
- 确保提交历史清晰（必要时使用 `git rebase -i` 整理）

</Step>

<Step>
### 编写清晰的 PR 描述

- 使用 PR 模板填写完整信息
- 说明**为什么**做这个改动，而不仅仅是**做了什么**
- 添加截图或 GIF 展示 UI 变化
- 列出测试步骤
- 关联相关的 Issue

</Step>

<Step>
### 保持 PR 大小合理

- 单个 PR 的改动应该在 **500 行以内**
- 如果改动较大，考虑拆分成多个 PR
- 每个 PR 应该有明确的主题

</Step>

<Step>
### 及时响应审查意见

- 认真对待每一条审查意见
- 如果不同意某个意见，礼貌地说明理由
- 修改后及时通知审查者
- 不要在 PR 中进行无关的改动

</Step>
</Steps>

### 代码审查指南

作为**审查者**：

<Tabs items={['关注点', '审查清单', '反馈技巧']}>
  <Tab value="关注点">
**代码质量**：代码是否清晰易读、是否遵循项目的编码规范、是否有适当的注释、是否有重复代码

**功能正确性**：代码是否实现了预期功能、是否处理了边界情况、是否有潜在的 bug

**测试覆盖**：是否有足够的单元测试、测试是否覆盖了主要场景、是否有 E2E 测试（如果需要）

**性能和安全**：是否有性能问题、是否有安全隐患、是否有内存泄漏风险

  </Tab>

  <Tab value="审查清单">
**检查项**：

✓ 代码可读且有良好的文档
✓ 更改经过了适当的测试
✓ 没有明显的性能问题
✓ 已解决安全考虑
✓ 破坏性变更已得到适当记录
✓ 提交信息符合规范
✓ 没有引入新的 ESLint 警告
